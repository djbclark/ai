Where things stand

The suite is green — 80 tests pass — because almost nothing here is a crash. Every finding below is quietly wrong output: a number that looks plausible, a row that vanishes with no error, an alert that never fires. That's the harder failure mode to catch and exactly what this review was built to surface.

Two threads run through the confirmed findings and tie directly to what you asked about:

  • Silent data loss compounds. A JSON-parsing fallback that can return the wrong fragment, a single 120-second call that has no partial-result path, and a selection rule that drops tokscale's Claude rows entirely — each on its own is a minor gap, but together they mean the report can look complete while several sources have quietly gone dark.
  • The scoring model has a self-cancelling formula at its center. One line of arithmetic that reduces to a constant is why Claude's 5-hour window tops your report run after run — not a tuning problem, a bug with one specific, traceable cause. See §2.

Critical — silently wrong, high blast radius

All live Claude data disappears when cswap is enabled but returns nothing
collectors/runner.py:98

_select_and_cross_check treats cswap as the sole authority for the claude provider whenever it's enabled — even if cswap produced zero rows. CodexBar's live Claude data is cross-checked but never promoted to fill the gap, and tokscale's Claude rows (which exist in a real run right now — Session 3% used, Weekly 64% used) are excluded from selection and from the cross-check, so the "no independent cross-check available" warning it does emit is itself inaccurate.

A cancelled-out formula pins Claude's 5-hour urgency at 100, every run
analysis/use_or_lose.py:153

cycles_needed = max(1, round((remaining/100) * capacity / capacity)) — capacity cancels algebraically. The expression is just round(remaining/100), which is always 1 for any remaining value under 150%. For Claude's 5-hour window (configured flexibility: 0.0), that forces earliest_start_calendar = resets_at − 300min, which is always in the past, which pins flexibility_urgency at exactly 100 on every single run, regardless of actual usage. This is the direct, traceable mechanism behind the behavior you flagged — full write-up in §2.

Reported "monthly waste" is off by more than 10×
report.py:380

_throttled_waste_line computes monthly_waste = value_usd * 30, treating a per-cycle dollar figure as a per-day figure. For a 5-hour window there are roughly 4.9 waking cycles per day, not per month — the correct multiplier is closer to value_usd × cycles_per_day × 30. As written, every throttled-waste estimate the tool prints overstates the real monthly cost by roughly 15×.

Monthly windows report more dollar value than the plan costs

_compute_value_at_risk divides monthly price by active_cycles = waking_minutes_per_month / window_minutes. For a monthly-bucketed window that's <1 active cycle, so value_per_refill > monthly_price. Previously latent because tokscale never set window_minutes; the uncommitted diff's new _infer_window_minutes makes this newly user-visible for every Copilot row.

A single 120-second tokscale call has no partial-result path
collectors/tokscale.py:25

This is the crux of your first ask. CodexBar solved exactly this problem — discover enabled providers via a fast local call, then query each one in its own concurrent subprocess (90s each) so one slow provider can't take the others down with it. tokscale gets none of that: one bundled call, one 120s timeout (not even matched to CodexBar's own 180s bundled-fallback budget), and if it times out every tokscale-sourced provider vanishes at once with a single generic error line. Full design in §3.

Design: fixing the rating algorithm

Your read was correct, and it traces to one specific bug plus one structural gap — not a tuning problem.

Why the 5-hour window always wins today

Claude's 5-hour and weekly windows draw from the same underlying allotment — burning the 5-hour window spends weekly budget. Three things in analysis/use_or_lose.py conspire to ignore that:

  1. Deadline urgency is absolute, not relative. days ≤ 0.5 → raw urgency 100. A 5-hour window is always within half a day of reset, so it always maxes this term — the term can't tell the difference between "resets in 5 hours because it's a 5-hour window" and "resets in 5 hours because something's actually urgent."
  2. The cancelled-out formula from §1. Claude's 5-hour override sets flexibility: 0.0; the cycles-needed bug then always computes 1, which always places the "earliest reasonable start time" in the past, which pins flexibility_urgency at exactly 100.
  3. Nothing looks at sibling windows. Each window is scored in isolation. The weekly window's actual state — 64% used with two days left — never factors into the 5-hour window's score at all.

Verified against your live numbers (Session 3% used/resets ~5h; Weekly 64% used/resets ~2d): at your configured Max pricing, the 5-hour window scores MEDIUM (72) and is the only alert printed — the weekly is filtered out entirely by the min_remaining_percent=40 gate before it's even scored (36% remaining < 40).

Recommendation Replace absolute-days deadline scoring with pace-based projection, and make short windows within a shared allotment report through their governing (longest) window instead of scoring independently.

The core idea: score pace, not deadline

For any window with a known reset time and duration, compute elapsed fraction (how far through the cycle you are) against used fraction (how much you've burned). Project forward at the current pace — blended with a learned historical burn rate when available — to get projected waste at reset. Rank by projected wasted dollars, not by how soon the window happens to close.

Three states replace one urgency ladder

  • Burn — projected waste at reset ≥ 30% of the window's value. This is today's alert, but ranked by real projected dollars instead of by raw remaining-%.
  • Conserve — new. At current pace you'll be locked out before reset. Replaces today's silent min_remaining_percent gate with an actual advisory: "pace yourself, don't burn 5-hour sessions, this resets Thursday."
  • On pace — no alert, matching expected consumption. Still visible in the per-window detail line so nothing looks broken, just quiet.

Shared-allotment gating — this is what stops the 5h window from winning

For an account where multiple window durations share one real budget (Claude and OpenCode Go, by config), the longest window with usable data governs. Shorter sibling windows (the 5-hour) never emit their own alert — they only contribute burn logistics to the governing alert ("about 14 five-hour sessions' worth of headroom, start by Tuesday 9am"), and when the governing window is in Conserve state, the message explicitly warns against burning the short window. A lone 5-hour window with no weekly sibling still scores normally — this only suppresses the redundant, misleading half of a coupled pair.

03

Design: containing tokscale's timeouts

The honest constraint first: tokscale's CLI has no per-provider flag today. tokscale usage --help exposes only --json, --light, --home — confirmed both from the help text and by inspecting the binary's compiled string table. CodexBar's pattern (discover enabled providers, fan out one subprocess per provider) can't be replicated verbatim until that changes upstream.

tokscale usage --provider · not available tokscale usage --json · warm run ~1s PATH shim: exec npx tokscale@latest

What's actually causing the risk, concretely

  • The tokscale on your PATH is a one-line shim — exec npx tokscale@latest "$@". A cold npx resolution (observed today: it auto-installed tokscale@4.7.0) runs inside the collector's 120-second timeout budget, and @latest means the exact binary running on any given day is unpinned.
  • One bundled call means one slow or hanging provider inside tokscale's own serial querying takes down every provider it reports — Copilot, Grok, and Claude's tokscale-side reading alike — with a single generic error line and no partial data.
  • None of this shows up in normal use: a warm run today completed in ~1 second. The risk is specifically cold-start latency and any one upstream provider API stalling.
Recommendation — do now, no upstream dependency Pin the tokscale version the shim resolves (stop trusting @latest inside a timed collector), raise the bundled timeout to match CodexBar's own 180s bundled-fallback budget, and demote tokscale from "must succeed" to cross-check-only: if it fails, CodexBar's live data for the same providers still populates the report, exactly as the runner's existing selection logic (runner.py:111) already prefers CodexBar when both are live. A tokscale failure should degrade the cross-check, never the primary report.

What true per-provider containment would require

Two real paths, in order of how much they depend on tokscale changing upstream:

  1. An upstream --provider flag on tokscale usage. Once it exists, collectors/tokscale.py can mirror codexbar.py exactly: a fast local discovery call, then one concurrent subprocess per provider with its own timeout and its own error isolation, falling back to today's single bundled call when an older tokscale version doesn't support the flag.
  2. Per-integration subcommands as a stopgap. tokscale already exposes separate codex, cursor, antigravity, trae, and warp subcommands for some integrations — worth checking by hand whether any of these expose the same usage data tokscale usage aggregates, which would let today's version get partial per-provider isolation without waiting on an upstream change. This wasn't confirmed hands-on before this review's usage budget ran out — flagging it as the one open question rather than a settled recommendation.

Given tokscale is already the secondary source for every provider it shares with CodexBar (runner.py only selects it when CodexBar has nothing live), the pragmatic sequencing is: ship the containment fixes above immediately, and treat true per-provider fan-out as a follow-up gated on confirming subcommand (2) or landing a flag request upstream — not a blocker to closing out the timeout risk this week.
